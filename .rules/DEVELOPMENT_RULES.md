# 项目开发规则

本文档定义了 Structure Boot 项目的开发规范和流程，作为 AI 提示词工程的基础，确保所有开发活动都遵循统一的标准。

---

## 1. 版本号管理规范

### 1.1 版本号格式

项目采用语义化版本控制（Semantic Versioning）：
```
MAJOR.MINOR.PATCH
```

- **MAJOR（主版本号）**：不兼容的 API 变更，破坏性更新
- **MINOR（次版本号）**：向后兼容的功能新增或改进
- **PATCH（修订号）**：向后兼容的 bug 修复

### 1.2 版本号使用规范

#### 1.2.1 代码配置中的版本号

**代码、配置文件、Shell脚本中必须使用快照版本号：**

- **pom.xml**：`<revision>1.3.5-SNAPSHOT</revision>`
- **其他配置文件**：使用 `-SNAPSHOT` 后缀标识开发版本
- **Shell 脚本**：版本变量使用快照版本号

**原因**：快照版本号会在发布时由 Maven 自动替换为正式版本号，避免手动维护多个版本号。

#### 1.2.2 文档中的版本号

**文档、示例代码、README 中必须使用正式版本号：**

- **README.md**：示例配置使用 `<structure.version>1.3.5</structure.version>`
- **CHANGELOG.md**：记录正式发布版本，如 `[1.3.5]`
- **API 文档**：使用正式版本号说明

**原因**：文档面向用户，应展示最终发布版本，避免用户混淆。

### 1.3 版本更新流程

#### 1.3.1 版本号生成规则

**AI 在进行任何代码修改前，必须遵循以下流程：**

1. **询问用户确认**：首先询问用户是否需要更新版本号
   ```
   提示词：当前项目版本为 {current_version}（快照版本），是否需要更新版本号？(是/否)
   ```

2. **获取新版本号**：若用户确认更新，引导用户输入新的版本号
   ```
   提示词：请输入新的版本号（格式：x.y.z，将自动添加 -SNAPSHOT 后缀）：
   ```

3. **获取更新内容**：要求用户提供本次更新的简要说明
   ```
   提示词：请简要描述本次更新的内容：
   ```

4. **验证版本号格式**：确保输入的版本号符合语义化版本控制规范

5. **记录更新履历**：将版本号和更新内容记录到 CHANGELOG.md（使用正式版本号）

#### 1.3.2 版本号递增规则

| 变更类型 | 版本号更新 | 代码配置（快照） | 文档记录（正式） |
|---------|-----------|----------------|----------------|
| 修复小问题 | PATCH + 1 | 1.3.5-SNAPSHOT | 1.3.5 |
| 添加新功能 | MINOR + 1, PATCH = 0 | 1.4.0-SNAPSHOT | 1.4.0 |
| 破坏性变更 | MAJOR + 1, MINOR = 0, PATCH = 0 | 2.0.0-SNAPSHOT | 2.0.0 |

## 2. 修改规范

### 2.1 修改前准备

**AI 必须执行以下步骤：**

1. **确认修改范围**：明确需要修改的文件和功能
2. **检查版本号**：确认当前项目版本号（代码配置为快照版本）
3. **询问版本更新**：按照 1.3.1 流程询问用户是否更新版本

### 2.2 修改执行

**修改过程中必须遵循：**

1. **最小化变更**：只修改必要的代码，避免无关改动
2. **代码风格一致**：遵循项目现有的代码风格和命名规范
3. **添加必要注释**：对复杂逻辑添加清晰的注释说明
4. **保持兼容性**：优先保证向后兼容
5. **版本号一致性**：代码配置使用快照版本（如 `1.3.5-SNAPSHOT`），文档使用正式版本（如 `1.3.5`）

### 2.3 修改完成后

**AI 必须完成以下操作：**

1. **更新版本号**：在 pom.xml 等配置文件中更新为快照版本号（自动添加 -SNAPSHOT 后缀）
2. **更新变更日志**：在 CHANGELOG.md 中使用正式版本号记录本次修改
3. **同步文档版本**：更新 README.md、示例代码中的正式版本号
4. **验证修改**：检查修改是否正确，确保项目能正常编译
5. **提交说明**：提供清晰的修改摘要

## 3. 变更日志规范

### 3.1 日志格式

每条变更记录必须包含以下信息：

```markdown
## [版本号] - YYYY-MM-DD

### 变更类型

- 简短清晰的变更说明（不超过 50 字）
- 涉及的模块或文件
- 重要的 API 变更说明（如有）
```

### 3.2 变更类型分类

| 分类 | 说明 | 示例 |
|-----|------|------|
| **功能新增** | 添加新功能或模块 | 新增 Redis 分布式锁支持 |
| **功能增强** | 现有功能的改进 | 优化日志记录性能 |
| **Bug 修复** | 修复已知问题 | 修复分页查询 bug |
| **配置更新** | 更新配置文件 | 更新 JDK 版本配置 |
| **文档更新** | 更新文档内容 | 更新 API 文档 |
| **依赖升级** | 升级第三方依赖 | 升级 Spring Boot 版本 |

### 3.3 日志编写要求

1. **简洁明了**：每条记录不超过 2 行
2. **使用动词开头**：修复、新增、优化、更新、升级等
3. **明确范围**：指明影响的模块或功能
4. **时间准确**：使用实际修改日期

## 4. AI 提示词模板

### 4.1 版本更新提示词

```

1. 新版本号（格式：x.y.z，代码配置将自动添加 -SNAPSHOT 后缀）：
2. 本次更新内容简介：
```

### 4.2 修改完成提示词

```
修改已完成！

版本号更新：
- 代码配置（快照版本）：{old_version} → {new_version}-SNAPSHOT
- 文档记录（正式版本）：{old_version} → {new_version}

更新内容：
{change_summary}

已更新文件：
- pom.xml 配置文件（快照版本）
- CHANGELOG.md（正式版本）
- README.md 及其他文档（正式版本）
- {modified_files}

已记录变更日志至 CHANGELOG.md
```

### 4.3 错误处理提示词

```
版本号格式错误！

请输入符合语义化版本控制的版本号（格式：x.y.z，其中 x、y、z 为非负整数）

例如：1.3.5、2.0.0、1.4.0-SNAPSHOT
```

## 5. 开发流程规范

### 5.1 开发分支管理

- `main`：主分支，稳定版本
- `develop`：开发分支，日常开发
- `feature/*`：功能分支，新功能开发
- `hotfix/*`：修复分支，紧急 bug 修复

### 5.2 代码提交规范

提交信息格式：
```
[类型] 简短描述（不超过 50 字）

详细描述（可选）
```

类型说明：
- `feat`：新功能
- `fix`：bug 修复
- `docs`：文档更新
- `style`：代码格式
- `refactor`：代码重构
- `test`：测试代码
- `chore`：构建/工具更新

### 5.3 代码审查要求

1. 所有代码修改必须经过代码审查
2. 审查重点：代码质量、安全性、性能、兼容性
3. 审查通过后才能合并到主分支

---

## 6. 注意事项

1. **版本号唯一性**：每个版本号只能使用一次，不能重复发布
2. **SNAPSHOT 版本**：开发中可使用 `-SNAPSHOT` 后缀标识开发版本
3. **回滚策略**：如发现严重问题，应发布新的修复版本，而非回滚
4. **通知机制**：重大版本更新应通知相关依赖方
