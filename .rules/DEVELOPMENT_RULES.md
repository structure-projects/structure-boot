# Structure Boot AI 开发助手规范

本文档是 Structure Boot 项目的 AI 开发助手行为规范，定义了 AI 在协助开发时必须遵循的规则、提示词模板和检查清单。

---

## 1. 核心原则

### 1.1 基本行为准则

1. **保守修改原则**：优先最小化变更，只修改必要的代码
2. **向后兼容原则**：优先保证向后兼容，不做破坏性更改
3. **用户确认原则**：涉及版本更新、删除文件等重大操作前必须询问用户
4. **版本一致性原则**：代码配置用快照版本（-SNAPSHOT），文档用正式版本
5. **文档同步原则**：代码变更后同步更新相关文档

### 1.2 禁止行为

- 禁止在未询问用户的情况下删除文件
- 禁止在未询问用户的情况下修改版本号
- 禁止提交包含 TODO/FIXME 但未完成的代码
- 禁止添加空实现的代码（占位代码）
- 禁止添加未使用的依赖

## 2. 版本号管理

### 2.1 版本号规范

| 场景 | 版本格式 | 示例 |
| :--- | :--- | :--- |
| pom.xml 配置 | `x.y.z-SNAPSHOT` | `1.4.2-SNAPSHOT` |
| 文档/示例 | `x.y.z` | `1.4.2` |
| Git tag | `v.x.y.z` | `v1.4.2` |

### 2.2 版本号递增规则

| 变更类型 | 版本更新 | 代码配置 | 文档 |
| :--- | :--- | :--- | :--- |
| Bug 修复 | PATCH + 1 | `1.4.2-SNAPSHOT` | `1.4.2` |
| 新功能 | MINOR + 1, PATCH = 0 | `1.5.0-SNAPSHOT` | `1.5.0` |
| 破坏性变更 | MAJOR + 1 | `2.0.0-SNAPSHOT` | `2.0.0` |

### 2.3 版本更新流程

**当需要进行代码修改时，AI 必须遵循以下流程：**

```
1. 确认当前版本
   - 读取 pom.xml 中的 <revision> 标签
   - 确认当前为快照版本

2. 询问用户是否需要更新版本
   - "当前项目版本为 {version}（快照版本），是否需要更新版本号？(是/否)"

3. 若用户确认更新，获取：
   - 新版本号（格式：x.y.z）
   - 变更内容简介

4. 执行修改

5. 更新变更日志
```

## 3. 文件操作规范

### 3.1 文件创建

**创建新文件前必须检查：**

```markdown
1. 文件是否已存在（避免重复创建）
2. 文件是否符合项目目录结构
3. 是否需要添加版权声明
4. 是否需要添加 Javadoc 注释
```

**新建 Java 类必须包含：**

```java
/*
 * Copyright (c) 2025 Structure Boot
 *
 * Licensed under the Apache License, Version 2.0 (the "License");
 * ...
 */

package cn.structured.{module}.{layer};

/**
 * 类功能描述
 *
 * @author chuck
 * @version 1.0.0
 * @since 2026-06-23
 */
public class Xxx {
}
```

### 3.2 文件修改

**修改文件前必须检查：**

```markdown
1. 是否会影响其他模块（检查依赖关系）
2. 是否保持向后兼容
3. 是否需要更新版本号
4. 是否需要更新文档
5. 是否需要更新 CHANGELOG
```

**修改时遵循：**

- 最小化变更：只修改必要的代码
- 保持风格一致：遵循项目现有的代码风格
- 添加必要注释：对复杂逻辑添加清晰的注释

### 3.3 文件删除

**删除文件前必须：**

```markdown
1. 确认文件是否被其他模块引用
2. 询问用户确认删除
3. 检查是否需要从 pom.xml 中移除依赖
4. 检查是否需要更新相关文档
```

## 4. 代码风格规范

### 4.1 命名规范

| 类型 | 规范 | 示例 |
| :--- | :--- | :--- |
| 包名 | 全小写，点分隔 | `cn.structured.common.entity` |
| 类名 | PascalCase | `UserController`, `DateUtil` |
| 方法名 | camelCase | `getUserById`, `saveOrder` |
| 变量名 | camelCase | `userName`, `orderList` |
| 常量 | 全大写，下划线 | `MAX_RETRY_COUNT` |

### 4.2 代码格式

```java
// ✅ 正确格式
public class Example {

    private static final String CONSTANT = "value";

    public void method(String param) {
        if (param != null) {
            doSomething();
        }
    }
}

// ❌ 错误格式
public class Example{
    private static final String CONSTANT="value";
    public void method(String param){
    if(param!=null){
    doSomething();
    }
    }
}
```

### 4.3 缩进和空格

- 使用 4 空格缩进
- 操作符前后加空格：`a + b`
- 逗号后加空格：`method(a, b, c)`
- 方法括号不紧贴内容：`method(a, b)`

## 5. 任务场景提示词

### 5.1 Bug 修复场景

**问题诊断提示词：**

```
【问题诊断】
请分析以下错误信息：

错误类型：{error_type}
错误信息：{error_message}
相关文件：{file_path}
堆栈跟踪：{stack_trace}

请定位问题原因，并提供修复方案。
```

**修复完成提示词：**

```
【Bug 修复完成】
修复内容：{bug_description}
修复方案：{fix_approach}
涉及文件：{modified_files}

是否需要更新 CHANGELOG？(是/否)
```

### 5.2 功能开发场景

**功能开发提示词：**

```
【功能开发】
模块：{module_name}
功能名称：{feature_name}
功能描述：{feature_description}

请按照以下步骤开发：
1. 确认现有类似功能的实现方式
2. 遵循现有代码风格
3. 添加必要的注释和文档
4. 确保向后兼容
```

### 5.3 重构场景

**重构提示词：**

```
【代码重构】
重构目标：{refactor_goal}
涉及文件：{files}
原因说明：{reason}

重构原则：
1. 保持外部行为不变
2. 最小化变更范围
3. 确保测试通过
4. 更新相关文档

是否需要备份原代码？(是/否)
```

## 6. 变更日志规范

### 6.1 变更记录格式

```markdown
## [版本号] - YYYY-MM-DD

### 变更类型

- 变更描述（不超过 50 字）
- 涉及模块或文件
- 重要 API 变更（如有）
```

### 6.2 变更类型分类

| 类型 | 说明 | 示例 |
| :--- | :--- | :--- |
| `feat` | 新功能 | 新增 Redis 分布式锁支持 |
| `fix` | Bug 修复 | 修复分页查询问题 |
| `docs` | 文档更新 | 更新 API 文档 |
| `style` | 代码格式 | 格式化代码 |
| `refactor` | 重构 | 重构异常处理逻辑 |
| `perf` | 性能优化 | 优化缓存策略 |
| `test` | 测试 | 添加单元测试 |
| `chore` | 构建/工具 | 更新依赖版本 |

## 7. 检查清单

### 7.1 代码修改后检查

- [ ] 代码能正常编译
- [ ] 遵循命名规范
- [ ] 添加了必要的注释
- [ ] 没有 TODO/FIXME 遗留
- [ ] 没有空实现代码
- [ ] 没有未使用的导入
- [ ] 更改已记录到 CHANGELOG

### 7.2 文件创建后检查

- [ ] 包含版权声明
- [ ] 包含 Javadoc 注释
- [ ] 遵循项目代码风格
- [ ] 文件位置正确
- [ ] 已添加到对应模块的 pom.xml（如需要）

### 7.3 版本更新后检查

- [ ] pom.xml 版本号已更新
- [ ] CHANGELOG 已记录
- [ ] 文档版本号已同步
- [ ] 示例代码版本号已更新

## 8. 错误处理

### 8.1 版本号格式错误

```
版本号格式错误！

请输入符合语义化版本控制的版本号：
- 格式：x.y.z（x、y、z 为非负整数）
- 示例：1.4.2、2.0.0、1.5.0

注意：
- 快照版本需添加 -SNAPSHOT 后缀
- 文档和示例使用正式版本号
```

### 8.2 文件已存在

```
文件已存在！

文件路径：{file_path}
请选择操作：
1. 读取并修改现有文件
2. 使用其他文件名
3. 取消操作
```

### 8.3 文件被占用

```
文件可能被其他程序占用！

文件路径：{file_path}
建议：
1. 关闭可能占用该文件的程序
2. 确认文件是否正在被编辑
3. 稍后重试
```
