# AI 任务场景提示词

本文档定义了 Structure Boot 项目中常见任务场景的 AI 提示词模板，帮助 AI 在处理不同类型任务时遵循统一的规范和流程。

---

## 1. Bug 修复场景

### 1.1 问题诊断

```
【Bug 诊断任务】

请分析以下问题：

错误类型：{error_type}
错误信息：{error_message}
相关文件：{file_path}
堆栈跟踪：
{stack_trace}

诊断步骤：
1. 定位问题发生的准确位置
2. 分析问题产生的原因
3. 确认是否影响其他模块
4. 提出修复方案

请输出：
- 问题根因分析
- 修复方案（可选多个）
- 修复优先级
```

### 1.2 修复执行

```
【Bug 修复执行】

问题描述：{bug_description}
修复方案：{selected_fix}
涉及文件：{files_to_modify}

修复步骤：
1. 备份原文件（如需要）
2. 应用修复
3. 验证修复效果
4. 更新 CHANGELOG

修复完成标准：
- [ ] 代码能正常编译
- [ ] 原有测试通过
- [ ] CHANGELOG 已更新
```

### 1.3 回归测试

```
【回归测试】

请对修复的 Bug 进行回归测试：

修复模块：{module_name}
测试范围：
- 修复的功能点
- 相关依赖模块
- 边界条件

请确认：
1. 修复后的功能是否正常
2. 是否引入新的问题
3. 是否影响其他功能
```

## 2. 功能开发场景

### 2.1 新功能开发

```
【新功能开发任务】

模块：{module_name}
功能名称：{feature_name}
功能描述：{feature_description}

需求分析：
1. 确认功能的输入和输出
2. 确定功能边界
3. 分析对现有代码的影响
4. 确认向后兼容性

4. 添加单元测试
5. 更新相关文档

代码位置：
- 新增代码：{new_files}
- 修改代码：{modified_files}
```

功能名称：{feature_name}
现有功能：{current_behavior}
增强目标：{enhanced_behavior}

增强原则：
1. 保持向后兼容
2. 最小化变更
3. 添加配置项控制新行为（如需要）

实现步骤：
1. 分析现有实现
2. 确定增强点
3. 设计扩展方案
4. 实现增强功能
5. 添加测试
```

### 2.3 接口设计

```
【接口设计任务】

场景：{scenario}
接口类型：REST API / 内部接口 / 回调接口

接口定义：
- 接口名称：{interface_name}
- 请求参数：{request_params}
- 返回结果：{response_result}
- 异常处理：{exception_handling}

设计原则：
1. RESTful API 遵循 HTTP 方法语义
2. 参数命名清晰
3. 返回格式统一
4. 考虑版本演进

请确认接口设计是否合理？
```

## 3. 重构场景

### 3.1 代码重构

```
【代码重构任务】

重构目标：{refactor_goal}
涉及文件：{files}
代码问题：{current_issues}

重构原则：
1. 保持外部行为不变
2. 最小化变更范围
3. 确保测试覆盖
4. 分步骤执行

重构步骤：

2. 识别需要重构的代码
3. 设计重构方案
4. 小步前进，每次重构后验证
5. 更新测试（如需要）
6. 更新文档

风险评估：
- 影响范围：{impact_scope}
- 回滚难度：{rollback_difficulty}
```

任务类型：拆分 / 合并
目标模块：{target_module}

当前状态：
- 现有结构：{current_structure}
- 目标结构：{target_structure}

操作步骤：
1. 分析依赖关系
2. 制定迁移计划
3. 创建新模块结构
4. 迁移代码
5. 更新依赖配置
6. 验证编译

## 4. 依赖管理场景

### 4.1 依赖升级

```
【依赖升级任务】

依赖名称：{dependency_name}
当前版本：{current_version}
目标版本：{target_version}

升级分析：
1. 变更内容
2. 兼容性影响
3. 需要的代码调整
4. 测试重点

升级步骤：
1. 在 structure-dependencies/pom.xml 中更新版本

回滚方案：
如果升级失败，如何回滚到当前版本
```

### 4.2 新增依赖

```
【新增依赖任务】

依赖名称：{dependency_name}
依赖版本：{version}
用途说明：{purpose}

添加位置：
- structure-dependencies/pom.xml（统一版本管理）
- 或具体模块的 pom.xml（仅该模块使用）

添加步骤：
1. 确认依赖是否已存在（避免重复）
2. 确认版本是否已在 dependencies 中管理
3. 添加依赖
4. 验证编译
5. 确认许可证兼容（Apache License 2.0）
```

## 7. 安全相关场景

### 7.1 敏感信息处理

```
【敏感信息检查任务】

检查范围：{scope}

检查项：
1. 密码是否加密存储
2. 敏感配置是否外泄
3. 日志是否记录敏感信息
4. API 返回是否包含敏感数据

处理原则：
1. 密码必须加密
2. 敏感信息脱敏
3. 配置通过环境变量或加密配置
4. 日志记录时脱敏

请检查并修复发现的敏感信息泄露问题
```

### 7.2 SQL 注入防护

```
【SQL 注入检查任务】

检查范围：{scope}

检查项：
1. MyBatis #{} vs ${} 使用
2. 动态 SQL 安全性
3. 用户输入校验

修复原则：
1. 优先使用 #{}（参数绑定）
2. 避免使用 ${}（字符串替换）
3. 如必须使用 ${}，确保参数已校验
```

## 9. 迁移升级场景

### 9.1 Spring Boot 版本升级

```
【Spring Boot 版本升级任务】

当前版本：{current_version}
目标版本：{target_version}

升级步骤：
1. 在 structure-dependencies/pom.xml 更新 spring-boot.version
2. 逐个模块验证编译
3. 检查不兼容 API 的使用

## 10. 发布相关场景

### 10.1 版本发布

```
【版本发布任务】

发布版本：{version}
发布类型：正式版 / 快照版

发布前检查：
1. [ ] 所有测试通过
2. [ ] 代码审查已通过
3. [ ] CHANGELOG 已更新
4. [ ] 版本号已确认
5. [ ] 文档已就绪

发布步骤：
1. 更新 pom.xml 为正式版本（移除 -SNAPSHOT）
2. 提交并打标签
3. 执行发布脚本
4. 更新文档版本号
5. 通知相关方

发布后：
1. [ ] Maven Central 同步完成
2. [ ] GitHub Release 已创建
3. [ ] 文档已更新
```

### 10.2 热修复发布

```
【热修复发布任务】

问题描述：{bug_description}
影响版本：{affected_versions}
修复版本：{fix_version}

紧急修复流程：
1. 从受影响版本创建 hotfix 分支
2. 定位并修复问题
3. 测试验证
4. 合并到 main 和 develop
5. 发布修复版本

注意：
- 热修复应该最小化变更
- 只修复问题，不添加新功能
- 修复后需同步到主开发分支
```
