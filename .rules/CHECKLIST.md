# AI 自检清单

本文档是 Structure Boot 项目 AI 开发助手的自检清单，AI 在完成任务后应逐项检查，确保符合项目规范。

---

## 1. 代码质量检查

### 1.1 命名规范

- [ ] 包名全小写，使用点分隔符
- [ ] 类名使用 PascalCase
- [ ] 方法名使用 camelCase
- [ ] 变量名使用 camelCase
- [ ] 常量使用 UPPER_SNAKE_CASE
- [ ] 命名有意义，能表达实际含义

### 1.2 代码格式

- [ ] 使用 4 空格缩进
- [ ] 左括号 `{` 与语句在同一行
- [ ] 右括号 `}` 单独占一行
- [ ] 操作符前后有空格
- [ ] 逗号后有空格
- [ ] 行长度不超过 120 字符

### 1.3 代码结构

- [ ] 没有空实现的代码
- [ ] 没有 TODO/FIXME 遗留
- [ ] 没有未使用的导入
- [ [ ] 没有重复代码
- [ ] 遵循 SOLID 原则
- [ ] 工具类有私有构造方法

### 1.4 注释规范

- [ ] 版权声明完整（新建 Java 文件）
- [ ] 类有 Javadoc 注释
- [ ] 方法有 Javadoc 注释
- [ ] 复杂逻辑有行内注释
- [ ] 注释与代码同步更新

## 3. 安全检查

### 3.1 敏感信息

- [ ] 没有硬编码密码
- [ ] 没有硬编码密钥
- [ ] 敏感配置通过环境变量
- [ ] 日志中没有敏感信息
- [ ] API 返回数据已脱敏

### 3.2 SQL 注入

- [ ] 使用 #{} 而非 ${}
- [ ] 用户输入已校验
- [ ] 动态 SQL 已防护

### 3.3 其他安全

- [ ] 没有 SQL 注入风险
- [ ] 没有 XSS 风险（如涉及 Web）
- [ ] 权限校验适当

## 自检执行指南

### 执行时机

1. 完成代码修改后
2. 完成文件创建后
3. 完成版本更新后
4. 准备提交前

### 执行方式

```
完成修改后，逐项检查清单内容。

如发现问题：
1. 立即修复
2. 重新检查
3. 确认无误后继续

如有疑问：
1. 记录疑问点
2. 询问用户确认
3. 根据反馈处理
```

### 检查结果记录

```
【自检完成】

检查结果：✅ 全部通过 / ⚠️ 发现问题

发现问题：

2. {issue_2} - 已修复 / 待修复

是否需要用户确认：
- {items_requiring_user_confirmation}
```
